# 数据源抽象 - 前端对接说明

本文档说明在完成数据源抽象与 QMT 连接参数化接口后，前端需要同步完成的配置入口与同步任务发起交互的调整，确保与后端参数化设计一致。

## 一、数据源连接配置入口

### 1.1 新增页面与路由

- **页面**：数据源连接管理（如 `DataSourceConnections.vue` 或 `DataSourceConfig.vue`），用于对「数据源连接」进行增删查改。
- **路由**：建议路径 `/settings/data-sources` 或 `/data-sources`，名称如 `DataSourceConnections`；在 [frontend/src/router/index.js](frontend/src/router/index.js) 中注册，并在主导航或「设置」下增加入口（如「数据源」或「QMT 连接」）。

### 1.2 功能要点

- **列表**：调用 `GET /api/data-source/list`，展示连接名称、类型（qmt/joinquant/tushare）、是否启用、是否行情源、是否交易源等；支持按类型筛选。
- **新建**：表单包含名称、数据源类型（下拉，如 QMT/聚宽/tushare）、QMT 时必填：host、port、user、password、xt_quant_path、xt_quant_acct；以及「设为行情源」「设为交易驱动源」勾选。提交 `POST /api/data-source/connections`。
- **编辑**：进入编辑页或抽屉，回显后提交 `PUT /api/data-source/connections/{id}`。
- **删除**：确认后调用 `DELETE /api/data-source/connections/{id}`。
- **展示**：列表中可标识当前「行情源」「交易源」（与后端 `is_quote_source`、`is_trading_source` 对应），便于用户识别。

### 1.3 API 封装

- 在 `frontend/src/api/` 下新增 `dataSource.js`（或合并到现有 api 模块），封装上述接口：`getConnections(params)`、`createConnection(data)`、`updateConnection(id, data)`、`deleteConnection(id)`。
- 请求/响应字段与后端 Pydantic 及 `data_source_connections` 表一致。

## 二、同步任务发起交互的参数化调整

后端 `POST /api/security/update` 在参数化后增加可选参数：`source_type`（默认 `qmt`）、`source_id`（数据源连接 ID，可选）。前端发起证券同步任务时需传入上述参数。

### 2.1 证券列表页（SecurityList.vue）

- **当前行为**：「从 QMT 更新」「同步该板块」按钮调用 `securityApi.update(market, sector)`，未传数据源。
- **调整**：
  - 在页面加载时（或进入页时）请求可用数据源列表（如仅 `source_type=qmt` 且 `is_active=1`），用于下拉选项。
  - 「从 QMT 更新」：在弹窗或下拉中让用户选择「数据源」（展示连接名称，对应 `source_id`）；若不选则使用后端默认（等价于不传 `source_id`）。调用 `securityApi.update(market, sector, sourceType, sourceId)`。
  - 「同步该板块」：同样支持选择数据源后调用，参数包含 `sector` 与 `source_type`、`source_id`。
- **API 封装修改**：[frontend/src/api/security.js](frontend/src/api/security.js) 中 `update` 方法增加参数：`update(market = null, sector = null, sourceType = 'qmt', sourceId = null)`，在请求 params 中带上 `source_type`、`source_id`（有值时才传）。

### 2.2 板块页（Sectors.vue）

- **当前行为**：同步板块证券时使用 `fetch('/api/security/update', { method: 'POST', body: JSON.stringify({ sector: sectorName }) })`，且未传 `source_type`/`source_id`，且使用 body 而非 query。
- **调整**：
  - 改为与证券列表页一致的调用方式：使用封装好的 `securityApi.update(null, sectorName, sourceType, sourceId)`，或统一使用 `GET/POST` 的 query 参数（与后端约定一致：`POST /api/security/update?market=xx&sector=xx&source_type=qmt&source_id=1`）。
  - 若有「选择数据源」入口，可复用数据源列表与选择逻辑（如从 Pinia 或 api 获取连接列表，选一个再发起同步）。

### 2.3 交互细节建议

- **默认数据源**：若用户不选择，前端不传 `source_id`（或传后端约定的「默认」值），由后端使用默认 QMT 连接或第一条可用连接。
- **文案**：按钮可保留「从 QMT 更新」或改为「从数据源更新」，并在选择数据源时展示连接名称（如「本地 QMT」），避免与多数据源混淆。
- **错误与冲突**：任务冲突（409）或提交失败时，沿用现有提示逻辑；若后端返回「未配置数据源」等新错误，前端可增加对应提示。

## 三、与后端实现的对应关系

| 后端 | 前端 |
|------|------|
| `GET/POST /api/data-source/connections` 等 CRUD | 数据源连接管理页、api/dataSource.js |
| `POST /api/security/update?source_type=&source_id=` | securityApi.update(..., sourceType, sourceId)；SecurityList、Sectors 发起同步时传参 |
| 默认连接（未传 source_id） | 不选数据源时不传 source_id，或传空 |

完成上述前端改动后，数据源抽象与参数化在接口和交互上保持一致，后续扩展多数据源或切换行情/交易源时只需在配置页操作即可。
